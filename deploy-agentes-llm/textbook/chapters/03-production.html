Capítulo 03

Controles de Produção: Timeout, Concorrência e Resiliência

A diferença entre um prototipo e um serviço de produção não é o código do agente — são os controles ao redor dele.

Por que controles importam

A diferença entre um prototipo e um serviço de produção não é o código do agente — são os controles ao redor dele. Um agent.run() que funciona no notebook pode derrubar um sistema inteiro em produção se não houver limites claros de concorrência, timeout e tratamento de erro.

Neste capítulo, vamos construir cinco mecanismos essenciais para tornar um agente LLM operável em produção:

  • Concorrência controlada — limitar quantas chamadas ao LLM acontecem ao mesmo tempo
  • Timeout em camadas — cada nível da stack com seu proprio limite de tempo
  • Erros tipados — respostas HTTP específicas para cada tipo de falha
  • Health check — verificação automática da saúde do serviço
  • Logging estruturado — rastreabilidade por request com JSON
Princípio central

Cada um desses controles existe para proteger três coisas: o orçamento (tokens custam dinheiro), a disponibilidade (um request lento não pode travar os demais) é a debugabilidade (quando algo falha, você precisa saber onde e por que).

Semáforo de Concorrência

O asyncio.Semaphore funciona como uma sala de espera: define quantas requisições podem ser processadas simultaneamente. As demais aguardam na fila. Se a fila também encher, novas requisições são rejeitadas imediatamente com HTTP 429.

Para agentes LLM, controlar concorrência é ainda mais crítico do que em APIs tradicionais. Cada requisição consome tokens pagos do LLM provider, e providers como a Google (Gemini) impoe rate limits rigorosos. Sem um semáforo, 100 requisições simultâneas disparam 100 chamadas ao LLM — estourando limites e explodindo custos.

concurrency.py
from dataclasses import dataclass

@dataclass
class ConcurrencyStats:
    active: int
    waiting: int
    max_concurrent: int
    total_processed: int
    total_rejected: int

class AgentSemaphore:
    def __init__(self, max_concurrent=3, max_queue=10):
        self._semaphore = asyncio.Semaphore(max_concurrent)
        self._max_concurrent = max_concurrent
        self._max_queue = max_queue
        # Contadores atômicos (seguros em asyncio single-thread)
        self._active: int = 0
        self._waiting: int = 0
        self._total_processed: int = 0
        self._total_rejected: int = 0

    async def acquire(self):
        if self._waiting >= self._max_queue:
            self._total_rejected += 1
            raise ConcurrencyLimitError(
                f"Ativos: {self._active}/{self._max_concurrent}, "
                f"Fila: {self._waiting}/{self._max_queue}"
            )
        self._waiting += 1
        try:
            await self._semaphore.acquire()
        finally:
            self._waiting -= 1
        self._active += 1

    def release(self):
        self._active -= 1
        self._total_processed += 1
        self._semaphore.release()

    @property
    def stats(self) -> ConcurrencyStats:
        return ConcurrencyStats(
            active=self._active,
            waiting=self._waiting,
            max_concurrent=self._max_concurrent,
            total_processed=self._total_processed,
            total_rejected=self._total_rejected,
        )
Fila de processamento ≠ fila assíncrona

Quando falamos de "fila" no contexto do semáforo, estamos nos referindo a uma fila de processamento — um mecanismo de controle de concorrência que limita quantas requisições são processadas ao mesmo tempo. Essa fila existe tanto em sistemas síncronos quanto assíncronos. Não confundir com o padrão Async (capítulo anterior), onde a fila é um message broker (RabbitMQ) para processamento em background.

Atenção

O max_concurrent deve refletir o rate limit do seu LLM provider, não a capacidade de CPU do servidor. Um valor comum para APIs da Gemini e entre 3 e 5 chamadas simultâneas por instancia.

Timeout em Camadas

Um único timeout não é suficiente. Cada camada da stack tem latência diferente e precisa do seu proprio limite:

  • Request timeout (120s) — o tempo máximo que o cliente HTTP espera pela resposta completa. Protege o gateway é o load balancer.
  • Agent timeout (90s) — o tempo máximo para o agente completar todo o raciocínio (incluindo múltiplas tool calls). Protege contra loops infinitos do LLM.
  • Tool timeout (30s) — o tempo máximo para uma única tool call (busca no banco, API externa). Protege contra dependências travadas.

A propagação é de dentro para fora: se a tool excede 30s, o agente recebe um AgentTimeoutError e pode tentar uma estratégia alternativa. Se o agente excede 90s, o request retorna 504 Gateway Timeout ao cliente.

timeout_utils.py
async def with_timeout(coro, timeout_seconds, context="operação"):
    try:
        return await asyncio.wait_for(
            coro, timeout=timeout_seconds
        )
    except asyncio.TimeoutError:
        raise AgentTimeoutError(
            f"Timeout: {context} excedeu {timeout_seconds}s"
        )

Erros Tipados

Retornar HTTP 500 para toda falha impede o cliente de tomar decisões inteligentes. Se o erro é de timeout, o cliente pode tentar novamente. Se é de concorrência, deve esperar. Se a tool falhou, pode tentar sem aquela tool. Cada tipo de erro precisa de um código HTTP semântico é uma mensagem específicos.

Hierarquia de Erros

AgentServiceError # classe base
├── AgentTimeoutError → 504 Gateway Timeout
├── ConcurrencyLimitError → 429 Too Many Requests
├── ToolExecutionError → 502 Bad Gateway
└── AgentExecutionError → 500 Internal Server Error

Cada erro inclui um detail legível para o cliente é um error com o nome da classe para consumo programático. Isso permite que o frontend ou outro serviço implemente retry seletivo com lógica de backoff adequada.

Registrando os Handlers

A função register_error_handlers() instala um handler global no FastAPI que converte qualquer AgentServiceError em uma resposta JSON padronizada, independente do endpoint que lançou a exceção:

errors.py
def register_error_handlers(app: FastAPI) -> None:
    @app.exception_handler(AgentServiceError)
    async def _handle_agent_error(
        request: Request, exc: AgentServiceError
    ) -> JSONResponse:
        return JSONResponse(
            status_code=exc.status_code,
            content={
                "error": type(exc).__name__,
                "detail": exc.detail,
            },
        )

Retry Seletivo por Código HTTP

Com erros tipados, o cliente pode tomar decisões inteligentes sobre quando repetir a chamada:

Exceção HTTP Estratégia
ConcurrencyLimitError 429 Retry com backoff exponencial — servidor sobrecarregado, aguardar
AgentTimeoutError 504 Retry (agente pode estar sobrecarregado com outra requisição)
ToolExecutionError 502 Retry com limite — ferramenta externa pode estar temporariamente indisponível
AgentExecutionError 500 Não fazer retry — erro de lógica, retry não vai ajudar

Padrão try/except no Endpoint

O padrão correto no endpoint combina semáforo, timeout e tratamento de erro em um bloco único com finally garantindo o release():

main.py — endpoint /invoke
await agent_semaphore.acquire()
try:
    result = await with_timeout(
        agent_module.run(request.message),
        timeout_seconds=AGENT_TIMEOUT,
    )
except AgentServiceError:
    raise  # erros tipados já têm status HTTP correto
except Exception as exc:
    raise AgentExecutionError(detail=str(exc)) from exc
finally:
    agent_semaphore.release()

Health Check e Logging

Logging Estruturado

Em produção, print() não escala. Logs precisam ser em JSON com campos padronizados para permitir busca, filtragem e alertas automáticos. O campo mais importante é o request_id — ele conecta todos os logs de uma mesma requisição, desde a entrada até a resposta final.

O projeto usa structured logging via structlog, que emite JSON em produção e texto colorido em desenvolvimento — o mesmo código, dois modos:

logging_config.py
def setup_logging() -> None:
    shared_processors = [
        structlog.contextvars.merge_contextvars,
        structlog.stdlib.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.StackInfoRenderer(),
        structlog.processors.format_exc_info,
    ]

    if ENVIRONMENT == "production":
        renderer = structlog.processors.JSONRenderer()
    else:
        renderer = structlog.dev.ConsoleRenderer()  # cores no terminal

    structlog.configure(
        processors=[*shared_processors, renderer],
        wrapper_class=structlog.make_filtering_bound_logger(0),
        logger_factory=structlog.PrintLoggerFactory(),
        cache_logger_on_first_use=True,
    )

Middleware de Request ID

Um middleware HTTP injeta um request_id único em cada requisição e o propaga via structlog.contextvars — assim todos os logs emitidos durante aquela requisição carregam o mesmo ID automaticamente:

main.py — logging_middleware
@app.middleware("http")
async def logging_middleware(request, call_next):
    request_id = generate_request_id()
    structlog.contextvars.clear_contextvars()
    structlog.contextvars.bind_contextvars(request_id=request_id)

    start_time = time.monotonic()
    response = await call_next(request)
    duration_ms = int((time.monotonic() - start_time) * 1000)

    logger.info(
        "request_completed",
        method=request.method,
        path=request.url.path,
        status=response.status_code,
        duration_ms=duration_ms,
    )
    response.headers["X-Request-ID"] = request_id
    return response

Health Check

O orquestrador (Kubernetes, load balancer) precisa saber se o serviço está saudável. Um endpoint /health verifica conectividade com cada dependência crítica e retorna um status agregado — health check granular por componente:

health.py
async def run_health_check() -> dict:
    postgres  = check_postgres()    # SELECT 1 via SQLAlchemy
    rabbitmq  = check_rabbitmq()    # socket TCP porta 5672
    gemini    = await check_gemini()  # GET /v1beta/models (sem tokens)

    components = {
        "postgres": postgres,
        "rabbitmq": rabbitmq,
        "gemini": gemini,
    }
    all_ok = all(c["status"] == "ok" for c in components.values())

    return {
        "status": "healthy" if all_ok else "degraded",
        "components": components,
    }
Cuidado

O health check não deve chamar o LLM completo (custaria tokens). Basta verificar se a autenticação com o provider está funcionando — um GET /v1beta/models já é suficiente, custa zero tokens e confirma que a API key é válida e o serviço está acessível.

Resumo

Principais aprendizados

1. Concorrência controlada via semáforo protege contra estouro de rate limits e custos — o max_concurrent deve refletir o limite do LLM provider, não a capacidade do servidor.

2. Timeout em camadas (request > agente > tool) permite tratamento de erro granular — cada camada pode reagir de forma diferente quando o limite é atingido.

3. Erros tipados com codigos HTTP específicos permitem que clientes implementem retry inteligente e fallback seletivo.

4. Health check e logging estruturado com request_id são a base da observabilidade — sem eles, debugar problemas em produção é impossível.
