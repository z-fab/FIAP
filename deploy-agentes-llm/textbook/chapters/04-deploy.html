Capítulo 04

Deploy: Artesanal vs. Gerenciado

Compare estratégias de deploy — da VM com Docker Compose até plataformas gerenciadas — e saiba escolher o modelo certo para cada contexto.

Do Docker ao Mundo Real

Temos um serviço containerizado com 3 padrões de comunicação e controles de produção. A pergunta agora e: onde rodar isso?

Ao longo dos capítulos anteriores construímos, camada por camada, um agente LLM capaz de responder de forma síncrona, streamar tokens e processar requisições de forma assíncrona. Agora e hora de levar esse serviço para fora do seu laptop.

Ponto central

A resposta depende de quanto controle você precisa versus quanto trabalho quer fazer. Não existe estratégia universalmente superior — cada abordagem tem um contexto ideal.

As três grandes familias de deploy para servicos de agentes são:

  • Artesanal — você gerencia tudo, da VM ao processo.
  • Semi-gerenciado — a plataforma cuida do servidor; você cuida do container.
  • Gerenciado — a plataforma cuida de quase tudo; você entrega o código do agente.

Deploy Artesanal (Kubernetes em VM)

A abordagem mais direta: uma VM (EC2, Droplet, Compute Engine ou qualquer outro provedor) rodando um cluster Kubernetes. Você é responsável por tudo o que acontece dentro desse servidor.

Vantagens
  • Controle total sobre configuração e networking
  • Custo previsível — você paga a VM, não por request
  • Ferramentas familiares (Docker, SSH, systemd)
  • Sem vendor lock-in de plataforma
Desvantagens
  • Você gerencia scaling manualmente
  • Você é responsável por uptime, updates e backups
  • Single point of failure se a VM cair
  • Sem auto-healing de containers em falha

Quando usar

Equipes pequenas com tráfego previsível, projetos em fase de experimentação ou prototipagem, e times que preferem simplicidade operacional a automação.

Deploy Semi-Gerenciado (Cloud Run / ECS)

Plataformas como Google Cloud Run e AWS ECS/Fargate oferecem execução serverless de containers: você entrega a imagem Docker e a plataforma cuida do provisionamento de servidores, load balancing e auto-scaling.

Vantagens
  • Auto-scaling automático — de zero a N instâncias
  • Zero gerenciamento de servidor (sem SSH, sem patching)
  • Modelo pay-per-use: paga apenas quando há tráfego
  • Deploy com um único comando
Desvantagens
  • Cold starts — problemático para agentes com inicialização lenta
  • Menos controle sobre networking e egress
  • Vendor lock-in parcial nas APIs da plataforma
  • Limites de timeout (ex.: Cloud Run: default 5 min)

Atenção: timeout do Cloud Run

Cloud Run tem timeout máximo configurável até 60 minutos, mas o default é 5 minutos. Agentes que fazem cadeias longas de ferramentas podem estourar esse limite silenciosamente. Sempre ajuste --timeout no deploy e projete timeouts em camadas no proprio agente.

Plataformas Gerenciadas

Vertex AI Agent Engine (Google) e AWS Bedrock AgentCore são plataformas desenhadas especificamente para agentes LLM. Você entrega o código do agente; a plataforma assume responsabilidade pelo serving, scaling, monitoramento e versionamento.

Vantagens
  • Menor time-to-market — sem configurar infra
  • Observabilidade built-in (traces, métricas, logs)
  • Scaling automático otimizado para agentes
  • Versionamento e rollback nativos
Desvantagens
  • Menor controle sobre o ambiente de execução
  • Custo por request mais alto que infra propria
  • Vendor lock-in completo
  • Limitações nas customizações de runtime e networking

Trade-off fundamental

Quanto mais a plataforma faz por você, menos controle você tem. Plataformas gerenciadas são o extremo da conveniência — e também do lock-in. Avalie sempre o custo de migrar antes de adotar.
