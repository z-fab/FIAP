Padroes de Comunicacao: Sync, Stream, Async

Nao existe UM jeito certo de servir um agente. Existem tres padroes, e a escolha depende do caso de uso.

Visao Geral dos Padroes

Quando um cliente precisa se comunicar com um agente LLM, existem tres abordagens fundamentais. Cada uma faz trade-offs diferentes entre experiencia do usuario, complexidade de implementacao e escalabilidade.

Padrao Sincrono (Request-Response)

O padrao mais simples: o cliente envia uma requisicao HTTP, a API chama o agente, e a resposta so retorna quando o agente termina. A conexao aberta fica mantida durante todo o processamento.

API
Agent
LLM

Quando funciona bem

  • Agentes rapidos que respondem consistentemente em menos de 15 segundos
  • Integracao sistema-sistema (machine-to-machine) onde nao ha usuario esperando
  • Prototipacao e MVPs onde simplicidade e prioridade

Quando quebra: limites de timeout da infraestrutura

O problema do padrao sincrono nao esta no codigo — esta na infraestrutura entre o cliente e o servidor. Cada camada tem seu proprio timeout:

Componente Timeout Padrao Impacto
API Gateway (AWS) 29 segundos Hard limit, nao configuravel
ALB (AWS) 60 segundos Configuravel, mas com custo de conexao aberta
Cloud Run (GCP) 5 minutos Generoso, mas cobra por tempo de execucao
Nginx (default) 60 segundos proxy_read_timeout, configuravel

Atencao

Se o seu agente leva 35 segundos para responder e voce esta atras de um API Gateway da AWS, o cliente vai receber um 504 Gateway Timeout — mesmo que o agente esteja processando normalmente no backend.

Implementacao com FastAPI

endpoint_sync.py 
@app.post("/invoke", response_model=InvokeResponse)
async def invoke(request: InvokeRequest) -> InvokeResponse:
    result = await agent_module.run(request.message)
    return InvokeResponse(
        output=result.output,
        tools_used=result.tools_used,
        token_count=result.token_count,
        step_count=result.step_count,
        duration_ms=result.duration_ms,
    )

Padrao Streaming (SSE)

Server-Sent Events (SSE) e um protocolo HTTP que permite ao servidor enviar dados progressivamente ao cliente, sem que o cliente precise fazer polling. A conexao e aberta uma vez e o servidor envia eventos conforme o agente processa. O header de resposta usa text/event-stream como Content-Type.

Diferente de WebSocket, SSE e unidirecional (servidor para cliente) e funciona sobre HTTP/1.1 padrao — compativel com load balancers, proxies e CDNs sem configuracao especial.

Tipos de evento

Um agente LLM com tools produz uma sequencia previsivel de eventos. Definir tipos padronizados permite que o frontend reaja a cada etapa:

Evento Descricao Dados Tipicos
step_start Inicio de um step do agente {"step": 1, "type": "reasoning"}
tool_call Agente decidiu chamar uma tool {"tool": "search_database", "args": {...}}
tool_result Resultado da execucao da tool {"tool": "search_database", "result": "..."}
token Token individual da resposta final {"content": "A resposta"}
done Agente concluiu o processamento {"total_tokens": 1847, "steps": 3}
AG-UI: O padrao da industria

Os tipos de evento que definimos (step_start, tool_call, token, done) seguem o mesmo padrao que o protocolo AG-UI (Agent-User Interaction Protocol) formaliza para a industria.

Criado pela CopilotKit e adotado por LangGraph, Google ADK, AWS Bedrock e Microsoft, o AG-UI define ~30 tipos de evento padronizados para comunicacao agente-frontend via SSE. Nossos 5 eventos mapeiam diretamente para os equivalentes do AG-UI: step_start -> STEP_STARTED, tool_call -> TOOL_CALL_START, token -> TEXT_MESSAGE_CONTENT, done -> RUN_FINISHED.

O AG-UI adiciona granularidade extra (pares start/end para cada fase, sincronizacao de estado, eventos de raciocinio) e interoperabilidade entre frameworks — mas o conceito fundamental e exatamente o que estamos construindo aqui.

Implementacao com FastAPI + SSE

endpoint_stream.py 
from sse_starlette.sse import EventSourceResponse, ServerSentEvent

@app.post("/stream", response_class=EventSourceResponse)
async def stream(request: InvokeRequest):
    async for event in agent_module.run_stream(request.message):
        yield ServerSentEvent(
            event=event.type,
            data=event.data,
        )
Vantagem chave

Com streaming, o primeiro token pode chegar em menos de 1 segundo, mesmo que a resposta completa leve 30 segundos. Isso muda drasticamente a percepcao de velocidade pelo usuario.

Padrao Assincrono (Fila)

No padrao assincrono, o cliente envia a requisicao e recebe imediatamente um identificador de tarefa. O processamento acontece em background, em um worker separado. O cliente consulta o status periodicamente (polling) ou e notificado via webhook/websocket quando o resultado esta pronto. Este e o padrao fire-and-forget.

Componentes

  • Broker (RabbitMQ / Redis) — fila de mensagens que desacopla API do processamento
  • Worker (Celery / dramatiq) — processo que consome a fila e executa o agente
  • Persistencia (PostgreSQL) — armazena o status e resultado da tarefa

Implementacao com FastAPI + Celery

endpoint_async.py 
@app.post("/invoke/async", response_model=AsyncInvokeResponse)
async def invoke_async(request: InvokeRequest) -> AsyncInvokeResponse:
    record = TaskRecord(input_message=request.message, status="pending")
    db.add(record)
    db.commit()
    run_agent.delay(record.id, request.message)
    return AsyncInvokeResponse(
        task_id=record.id,
        status="pending",
        poll_url=f"/tasks/{record.id}",
    )

@app.get("/tasks/{task_id}")
async def get_task(task_id: str):
    record = db.get(task_id)
    return {
        "status": record.status,
        "result": record.result,
    }
Vantagem chave

O padrao assincrono e o unico que sobrevive a deploys, restarts e falhas de infra. Se o worker cair, a mensagem continua na fila e sera processada quando o worker voltar. Nenhum trabalho e perdido.

Use Sync quando...

  • O agente responde em menos de 15 segundos
  • A integracao e sistema-a-sistema (sem UI)
  • Simplicidade e mais importante que UX

Use Stream quando...

  • O usuario precisa ver progresso em tempo real
  • A resposta pode demorar 15-60 segundos
  • A experiencia interativa importa (chat, assistentes)

Use Async quando...

  • O processamento pode levar minutos
  • O cliente nao pode manter conexao aberta
  • Precisa absorver picos de trafego
  • Processamento em batch ou fire-and-forget

Resumo

Principais aprendizados

1. O padrao sincrono e simples mas fragil — qualquer timeout de infra pode matar a requisicao, mesmo que o agente esteja funcionando.

2. Streaming (SSE) muda drasticamente a percepcao de velocidade, permitindo feedback progressivo enquanto o agente processa.

3. O padrao assincrono e o mais resiliente, mas exige uma arquitetura mais complexa com broker, workers e persistencia.

4. A escolha nao e exclusiva — muitos sistemas oferecem os tres endpoints simultaneamente para diferentes casos de uso.
