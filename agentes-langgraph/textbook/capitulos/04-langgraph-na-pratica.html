Capítulo 04

LangGraph na Prática: Construindo Agentes com Grafos

Aplicar os conceitos teóricos construindo um agente ReAct e um workflow com LangGraph.

No capítulo anterior, exploramos o padrão ReAct e a arquitetura conceitual de agentes que raciocinam e agem em ciclos. Agora, a pergunta natural é: como implementar isso na prática, com controle total sobre o fluxo de execução?

Durante anos, o AgentExecutor do LangChain foi a forma padrão de orquestrar agentes. Ele funcionava como uma caixa preta: recebia um prompt, um modelo e ferramentas, e executava um loop binário — ou o modelo chamava uma ferramenta, ou ele parava. Para protótipos simples, era suficiente. Para sistemas de produção, era insuficiente. Não havia como inserir lógica de aprovação humana entre etapas, não havia como ramificar o fluxo condicionalmente, e o controle sobre o estado interno era mínimo.

A linguagem de composição LCEL (LangChain Expression Language) avançou nessa direção, mas só suportava grafos acíclicos dirigidos (DAGs). Agentes reais, no entanto, precisam de ciclos — o modelo chama uma ferramenta, recebe o resultado, raciocina novamente, e pode chamar outra ferramenta. Esse loop é a essência do padrão ReAct, e DAGs simplesmente não o representam.

LangGraph surgiu para resolver exatamente isso: uma biblioteca que modela fluxos de trabalho de agentes como grafos cíclicos com estado, oferecendo controle total sobre cada nó, cada aresta e cada transição. Com mais de 28.500 estrelas no GitHub e adoção por empresas como Klarna, Uber, J.P. Morgan e BlackRock, LangGraph tornou-se a infraestrutura padrão para agentes em produção.

Neste capítulo, vamos construir. Começaremos pelo modelo mental de grafos, passaremos pela anatomia do LangGraph, e terminaremos com dois projetos completos: um agente ReAct e um workflow condicional.

⚠️ Atenção — LangGraph v1

Este capítulo segue a API do LangGraph 1.0 (outubro 2025). Os primitivos do grafo (StateGraph, MessagesState, tools_condition) permanecem idênticos às versões anteriores. A principal mudança é que a API prebuilt para agentes passou a ser create_agent, importada de langchain.agents. Veja a seção "O que mudou no v1" ao final do capítulo.

4.1 Por que Grafos? O Modelo Mental de Nós, Arestas e Estado

Antes de abrir o editor de código, precisamos internalizar o modelo mental que fundamenta o LangGraph. Pense em qualquer processo de decisão complexo: há etapas (nós), transições entre elas (arestas) e informação que se acumula ao longo do caminho (estado).

Definição

Grafo dirigido: Estrutura composta por nós (vértices) e arestas (arcos direcionados). Cada aresta conecta um nó de origem a um nó de destino, definindo a direção do fluxo. Quando o grafo permite ciclos — ou seja, caminhos que retornam a um nó já visitado — chamamos de grafo cíclico dirigido.

No contexto de agentes de IA, essa abstração se traduz diretamente:

Conceito do GrafoSignificado no Agente
NóUma etapa de processamento: chamar o LLM, executar uma ferramenta, avaliar uma condição
Aresta fixaTransição determinística: “após executar a ferramenta, sempre volte ao modelo”
Aresta condicionalTransição dinâmica: “se o modelo pediu uma ferramenta, vá para o nó de ferramentas; senão, termine”
EstadoToda a informação acumulada: mensagens, resultados de ferramentas, decisões intermediárias

A vantagem do modelo de grafos sobre um loop simples é a expressividade: você pode representar pipelines lineares, árvores de decisão, loops de retry, fluxos paralelos e padrões de aprovação humana — tudo na mesma abstração.

Dica

Dica prática: Antes de escrever código, desenhe o grafo do seu agente em papel ou em uma ferramenta visual. Identifique os nós (o que cada etapa faz), as arestas (como as etapas se conectam) e o estado (que informação precisa fluir). Esse exercício evita retrabalho e torna o código significativamente mais claro.

4.2 LangGraph: O que Ele Resolve

LangGraph não é apenas uma biblioteca de grafos genérica. Ele foi projetado especificamente para resolver quatro problemas centrais de agentes em produção:

  1. Gerenciamento de estado: O estado do agente (mensagens, variáveis, contexto) é mantido de forma estruturada e tipada, com regras claras de como cada nó pode atualizá-lo.
  2. Memória e persistência: Através de checkpointing, o grafo pode ser pausado, persistido em banco de dados e retomado — essencial para agentes que interagem com humanos de forma assíncrona.
  3. Sessões e multi-tenancy: Cada conversa (thread) mantém seu próprio histórico de estado, permitindo múltiplos usuários simultâneos sem interferência.
  4. Execução de ferramentas orquestrada: Ferramentas são nós do grafo, não callbacks opacos. Você decide exatamente quando, como e sob quais condições uma ferramenta é executada.

Os dois pilares arquiteturais do LangGraph são:

  • Controlabilidade: Não há prompts escondidos, loops implícitos ou comportamento mágico. Cada decisão do fluxo é explícita no grafo.
  • Runtime de produção: Streaming token a token, estado persistente entre requisições, human-in-the-loop nativo e execução durável com garantias transacionais.
Definição

Modelo Pregel: LangGraph é inspirado no modelo Pregel (Google, 2010), usado para processar grafos em larga escala. A execução ocorre em super-steps: a cada passo, todos os nós elegíveis no mesmo nível executam em paralelo, produzem atualizações de estado, e o framework garante a aplicação transacional (“tudo ou nada”) dessas atualizações antes de avançar ao próximo super-step.

4.3 Anatomia de um Grafo LangGraph

Vamos dissecar cada componente fundamental.

4.3.1 StateGraph e a Definição de Estado

StateGraph é a classe central do LangGraph — é ela que orquestra todo o grafo. Ela recebe como parâmetro um schema de estado, tipicamente definido como um TypedDict (ou um modelo Pydantic). O TypedDict define a forma do estado: quais campos existem e seus tipos. O StateGraph, por sua vez, usa esse schema para validar e gerenciar as atualizações de estado durante a execução.

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]

graph = StateGraph(AgentState)  # StateGraph usa o TypedDict como schema

Cada campo do estado representa uma informação que o grafo mantém e que os nós podem ler e atualizar. O TypedDict define o que existe no estado; o StateGraph define como esse estado flui entre os nós.

4.3.2 Reducers: Como o Estado é Atualizado

Quando um nó retorna {"messages": [nova_msg]}, como esse valor é combinado com o estado existente? A resposta são os reducers. Sem um reducer, o comportamento padrão é simplesmente sobrescrever — o último nó a atualizar um campo vence, e o valor anterior é perdido.

operator.add é o reducer mais simples: ele concatena listas. Se o estado atual tem [a, b] e um nó retorna [c], o resultado será [a, b, c]. Útil para acumular itens ao longo da execução. Já add_messages é um reducer especializado para mensagens de chat: ele apenda novas mensagens à lista, mas se uma mensagem tem o mesmo ID de uma existente, ele a substitui em vez de duplicar. Isso é útil para editar ou corrigir mensagens anteriores.

É possível criar reducers customizados — qualquer função com assinatura (valor_atual, novo_valor) -> valor_merged funciona:

def keep_max(current: int, new: int) -> int:
    return max(current, new)

class State(TypedDict):
    best_score: Annotated[int, keep_max]

A sintaxe utiliza Annotated do módulo typing para associar o reducer ao campo:

from langgraph.graph import add_messages

class AgentState(TypedDict):
    messages: Annotated[list, add_messages]  # Apenda mensagens
    iteration_count: int  # Sem reducer: sobrescrito a cada atualização
Atenção

Se dois nós executarem em paralelo e ambos atualizarem o mesmo campo sem reducer, o último a ser processado vencerá, e o valor do outro será perdido. Use operator.add ou add_messages sempre que precisar acumular valores.

4.3.3 MessagesState: O Atalho para Agentes Conversacionais

Para o caso muito comum de agentes que mantêm um histórico de mensagens, o LangGraph oferece MessagesState, um estado pré-definido:

from langgraph.graph import MessagesState

# Equivalente a:
# class MessagesState(TypedDict):
#     messages: Annotated[list[AnyMessage], add_messages]

Você pode estendê-lo adicionando campos:

class MyAgentState(MessagesState):
    user_name: str
    approved: bool

4.3.4 Nós (Nodes)

Nós são funções Python — síncronas ou assíncronas — que recebem o estado atual e retornam um dicionário com as atualizações:

def assistant(state: AgentState) -> dict:
    """No que invoca o LLM."""
    response = model.invoke(state["messages"])
    return {"messages": [response]}

async def analyze(state: AgentState) -> dict:
    """No assincrono de analise."""
    result = await run_analysis(state["messages"])
    return {"current_step": "analysis_complete", "messages": [result]}

Os nós são registrados no grafo com add_node:

builder = StateGraph(AgentState)
builder.add_node("assistant", assistant)
builder.add_node("analyze", analyze)

4.3.5 Arestas (Edges)

Arestas definem o fluxo entre nós. Há dois tipos:

Arestas fixas — transições determinísticas:

builder.add_edge(START, "assistant")     # Comeca no assistant
builder.add_edge("tools", "assistant")   # Apos tools, volta ao assistant

Arestas condicionais — transições dinâmicas baseadas em uma função de roteamento:

def route_decision(state: AgentState) -> str:
    last_message = state["messages"][-1]
    if hasattr(last_message, "tool_calls") and last_message.tool_calls:
        return "tools"
    return END

builder.add_conditional_edges(
    "assistant",          # No de origem
    route_decision,       # Funcao de roteamento
    {"tools": "tools", END: END}  # Mapeamento de retorno para no destino
)

A função de roteamento pode retornar uma lista de strings para disparar execução paralela de múltiplos nós.

4.3.6 Compilação

Após definir nós e arestas, o grafo precisa ser compilado:

graph = builder.compile()

A compilação valida a estrutura (nós sem arestas, ciclos inválidos, etc.) e produz um objeto executável. A partir deste ponto, o grafo é imutável.

4.4 Construindo um Agente ReAct Passo a Passo

Chegou o momento de colocar a teoria em prática. Vamos construir um agente ReAct de duas formas: usando o atalho prebuilt e construindo manualmente.

4.4.1 Abordagem Prebuilt: Três Linhas

A forma mais rápida de criar um agente é com create_agent, que encapsula o loop ReAct completo:

from langchain.agents import create_agent

def check_weather(city: str) -> str:
    """Retorna a previsao do tempo para uma cidade."""
    forecasts = {"Sao Paulo": "25C, parcialmente nublado", "Rio": "32C, ensolarado"}
    return forecasts.get(city, "Dados nao disponiveis")

graph = create_agent(
    "anthropic:claude-3-7-sonnet-latest",
    tools=[check_weather],
    system_prompt="Voce e um assistente meteorologico prestativo. Responda em portugues."
)

# Executar
result = graph.invoke({
    "messages": [{"role": "user", "content": "Como esta o tempo em Sao Paulo?"}]
})
Dica

create_agent é a API padrão para agentes ReAct. Ela oferece middleware nativo, structured output e human-in-the-loop built-in. Quando precisar de controle total sobre o grafo — como lógica de retry ou roteamento complexo — use a construção manual.

4.4.2 Abordagem Manual: Controle Total

Agora vamos construir o mesmo agente “tijolo por tijolo”. O walkthrough interativo abaixo mostra cada etapa da construção manual:

Definição

tools_condition: Função de roteamento prebuilt que inspeciona a última mensagem do estado. Se ela contém tool_calls (ou seja, o LLM decidiu chamar uma ferramenta), retorna "tools". Caso contrário, retorna END. É o coração do ciclo ReAct: enquanto o modelo quiser usar ferramentas, o grafo continua ciclando; quando o modelo responde diretamente ao usuário, o grafo termina.

A execução segue este fluxo:

  1. START → assistant: O modelo recebe as mensagens e decide se precisa de uma ferramenta.
  2. assistant → tools (condicional): Se o modelo gerou tool_calls, o nó ToolNode executa as ferramentas.
  3. tools → assistant: O resultado da ferramenta volta ao modelo para raciocínio.
  4. assistant → END (condicional): Quando o modelo responde sem tool calls, o grafo termina.

4.5 Construindo Workflows com Grafos

Agentes ReAct são poderosos, mas nem todo problema requer um loop de raciocínio livre. Muitos fluxos de trabalho em produção seguem padrões mais estruturados: pipelines lineares (etapa A → B → C) ou grafos condicionais (avaliar → rotear → processar). O LangGraph suporta ambos.

4.5.1 Pipeline Linear

Considere um pipeline de processamento de texto: limpar → resumir → classificar.

from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class PipelineState(TypedDict):
    raw_text: str
    clean_text: str
    summary: str
    category: str

def clean(state: PipelineState) -> dict:
    """Remove caracteres especiais e normaliza o texto."""
    text = state["raw_text"].strip().lower()
    text = " ".join(text.split())
    return {"clean_text": text}

def summarize(state: PipelineState) -> dict:
    """Gera um resumo usando o LLM."""
    response = model.invoke(
        f"Resuma em uma frase: {state['clean_text']}"
    )
    return {"summary": response.content}

def classify(state: PipelineState) -> dict:
    """Classifica o texto em uma categoria."""
    response = model.invoke(
        f"Classifique em uma categoria (tech/business/science): {state['summary']}"
    )
    return {"category": response.content}

# Construir o pipeline
builder = StateGraph(PipelineState)
builder.add_node("clean", clean)
builder.add_node("summarize", summarize)
builder.add_node("classify", classify)

builder.add_edge(START, "clean")
builder.add_edge("clean", "summarize")
builder.add_edge("summarize", "classify")
builder.add_edge("classify", END)

pipeline = builder.compile()

# Executar
result = pipeline.invoke({"raw_text": "  A OpenAI anunciou o GPT-5...  "})
print(result["category"])

4.5.2 Grafo Condicional

Agora, um cenário mais realista: um sistema de avaliação que roteia documentos com base em uma pontuação.

class EvalState(TypedDict):
    document: str
    score: float
    decision: str
    feedback: str

def evaluate(state: EvalState) -> dict:
    """Avalia o documento e atribui uma pontuacao."""
    response = model.invoke(
        f"Avalie de 0 a 10 a qualidade deste texto. "
        f"Retorne APENAS o numero: {state['document']}"
    )
    return {"score": float(response.content.strip())}

def route_by_score(state: EvalState) -> str:
    """Roteia com base na pontuacao."""
    score = state["score"]
    if score >= 8:
        return "approve"
    elif score <= 4:
        return "reject"
    else:
        return "review"

def approve(state: EvalState) -> dict:
    return {"decision": "approved", "feedback": "Documento aprovado."}

def reject(state: EvalState) -> dict:
    return {"decision": "rejected", "feedback": "Documento nao atende criterios."}

def review(state: EvalState) -> dict:
    return {"decision": "needs_review", "feedback": "Encaminhado para revisao humana."}

# Construir o grafo condicional
builder = StateGraph(EvalState)
builder.add_node("evaluate", evaluate)
builder.add_node("approve", approve)
builder.add_node("reject", reject)
builder.add_node("review", review)

builder.add_edge(START, "evaluate")
builder.add_conditional_edges(
    "evaluate",
    route_by_score,
    {"approve": "approve", "reject": "reject", "review": "review"}
)
builder.add_edge("approve", END)
builder.add_edge("reject", END)
builder.add_edge("review", END)

eval_graph = builder.compile()

result = eval_graph.invoke({"document": "Artigo sobre transformers..."})
print(f"Decisao: {result['decision']} - {result['feedback']}")
Exemplo

Exemplo do mundo real: Empresas de seguros usam grafos condicionais semelhantes para rotear sinistros. O nó de avaliação analisa o valor e a complexidade; sinistros simples e de baixo valor são aprovados automaticamente; sinistros complexos são roteados para analistas humanos; sinistros com indicadores de fraude são encaminhados para investigação.

4.6 Checkpointing, Memória e Persistência de Estado

Em aplicações reais, agentes não existem em execuções isoladas. Um chatbot precisa lembrar do que o usuário disse há cinco mensagens. Um agente de workflow pode ser pausado por horas enquanto aguarda aprovação humana. O checkpointing é o mecanismo que viabiliza tudo isso.

4.6.1 Checkpointers Disponíveis

CheckpointerPersistenciaCaso de uso
InMemorySaverVolátil (memória RAM)Desenvolvimento e testes
SqliteSaverDisco localProtótipos e aplicações single-node
PostgresSaverBanco relacionalProdução, múltiplas instâncias

4.6.2 Memória Conversacional com Threads

Cada conversa é identificada por um thread_id. Todos os checkpoints de uma thread formam uma sequência ordenada do histórico de estados:

from langgraph.checkpoint.memory import InMemorySaver

memory = InMemorySaver()
graph = builder.compile(checkpointer=memory)

config = {"configurable": {"thread_id": "session-42"}}

# Turno 1
result1 = graph.invoke(
    {"messages": [{"role": "user", "content": "Meu nome e Ana."}]},
    config=config
)

# Turno 2 - mesmo thread_id, o agente lembra do contexto
result2 = graph.invoke(
    {"messages": [{"role": "user", "content": "Qual e o meu nome?"}]},
    config=config
)
# O agente respondera "Ana" porque o historico esta preservado
Atenção

Sem checkpointer, cada chamada a invoke é completamente isolada. O agente não terá nenhuma memória de interações anteriores. Se você está construindo qualquer aplicação conversacional, o checkpointer é obrigatório, não opcional.

4.6.3 Inspecionando o Estado

# Obter o estado atual da thread
snapshot = graph.get_state(config)
print(snapshot.values)        # Estado completo atual
print(snapshot.next)          # Proximo(s) no(s) a executar
print(snapshot.created_at)    # Timestamp do checkpoint
print(snapshot.parent_config) # Referencia ao checkpoint anterior

# Navegar pelo historico completo
for state in graph.get_state_history(config):
    print(f"{state.created_at}: {len(state.values['messages'])} mensagens")

# Atualizar o estado manualmente (util para correcoes)
graph.update_state(config, {"messages": [{"role": "user", "content": "Correcao..."}]})

4.6.4 Memory Store: Memória Cross-Thread

Enquanto o checkpointer mantém memória dentro de uma thread (uma conversa), o Memory Store permite compartilhar informações entre threads — por exemplo, preferências de um usuário que devem persistir entre sessões diferentes:

from langgraph.store.memory import InMemoryStore

store = InMemoryStore()

# Salvar uma preferencia do usuario (acessivel de qualquer thread)
store.put(
    namespace=("users", "ana"),
    key="preferences",
    value={"language": "pt-BR", "expertise": "ML Engineering"}
)

# Recuperar em qualquer thread futura
memories = store.search(namespace=("users", "ana"))
for mem in memories:
    print(mem.value)  # {"language": "pt-BR", "expertise": "ML Engineering"}

A distinção é fundamental:

  • Checkpointer (thread-scoped): “O que aconteceu nesta conversa?”
  • Memory Store (cross-thread): “O que sabemos sobre este usuario independente da conversa?”

4.7 Human-in-the-Loop: Quando o Agente Precisa de Aprovação Humana

Nem toda decisão deve ser automatizada. Transferências bancárias, envio de e-mails em nome do usuário, exclusão de dados — essas ações exigem confirmação humana. O LangGraph oferece suporte nativo a esse padrão.

4.7.1 Breakpoints Estáticos

A forma mais simples: pausar o grafo antes ou depois de um nó específico.

graph = builder.compile(
    checkpointer=memory,
    interrupt_before=["tools"]  # Pausa ANTES de executar ferramentas
)

# Primeira execucao - para antes de "tools"
result = graph.invoke(
    {"messages": [{"role": "user", "content": "Envie um e-mail para o CEO"}]},
    config=config
)

# Inspecionar o que o agente quer fazer
snapshot = graph.get_state(config)
print(snapshot.next)  # ("tools",)

# Se aprovado, continuar a execucao passando None como input
result = graph.invoke(None, config=config)

4.7.2 Interrupt Dinâmico (Recomendado para Produção)

Para cenários mais sofisticados, a função interrupt() permite pausar dentro de um nó, com lógica condicional:

from langgraph.types import interrupt, Command

def process_transfer(state):
    amount = state["amount"]
    recipient = state["recipient"]

    # So pedir aprovacao para valores altos
    if amount > 10000:
        decision = interrupt(
            f"Transferencia de R${amount:,.2f} para {recipient}. "
            f"Aprovar? (approve/reject)"
        )

        if decision != "approve":
            return {"status": "cancelled", "reason": "Rejeitado pelo usuario"}

    # Executar a transferencia
    execute_transfer(amount, recipient)
    return {"status": "completed"}

Para retomar a execução após a interrupção:

# O usuario analisa e decide aprovar
result = graph.invoke(
    Command(resume="approve"),
    config=config
)
Atenção

Regras importantes para interrupt():

1. Não envolva em try/except genérico: O interrupt() funciona levantando uma exceção especial internamente. Um except Exception capturaria essa exceção e quebraria o mecanismo.

2. Side effects antes do interrupt devem ser idempotentes: Quando a execução é retomada, o nó é re-executado desde o início. Se você escreveu no banco de dados antes do interrupt(), essa escrita acontecerá novamente. Use operações idempotentes (e.g., INSERT ... ON CONFLICT DO NOTHING).

Resumo

Neste capítulo, saímos da teoria e construímos agentes funcionais com LangGraph. Os conceitos-chave foram:

  1. Grafos como modelo mental: Nós representam etapas de processamento, arestas representam transições, e o estado flui e se acumula ao longo da execução. Grafos cíclicos permitem representar o loop ReAct que DAGs não conseguem expressar.
  2. Anatomia do LangGraph: StateGraph como classe central, reducers controlando como o estado é atualizado (add_messages para conversações, operator.add para acumulação), nós como funções Python, e arestas fixas ou condicionais para definir o fluxo.
  3. Agente ReAct: Implementável em três linhas com create_agent (de langchain.agents) ou com controle total usando StateGraph + nós + tools_condition. O ciclo assistant → tools → assistant é o padrão fundamental.
  4. Workflows estruturados: Pipelines lineares para processamento sequencial e grafos condicionais para roteamento baseado em dados — nem todo problema requer um agente autônomo.
  5. Checkpointing e memória: InMemorySaver/PostgresSaver para persistência de estado entre turnos de conversa (thread-scoped), InMemoryStore para memória cross-thread.
  6. Human-in-the-loop: interrupt() como mecanismo de produção para pausar o agente e aguardar decisão humana, com Command(resume=) para retomar.

O que mudou no LangGraph v1

O LangGraph 1.0 foi lançado em outubro de 2025 como a primeira versão estável do framework. A filosofia central foi "estabilidade, não ruptura" — os primitivos do grafo foram mantidos idênticos, e a maioria das mudanças são deprecações (não remoções imediatas).

Primitivos preservados

StateGraph, MessagesState, tools_condition, interrupt(), Command — todos funcionam exatamente igual. Código que usa esses primitivos diretamente não precisa de alteração.
