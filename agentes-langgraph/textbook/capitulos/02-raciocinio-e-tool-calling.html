Capítulo 02

Raciocínio e Tool Calling: Como LLMs "Pensam" e "Agem"

Desmistificar como LLMs raciocinam (tokens intermediários) e chamam ferramentas (sinalização, não execução).

No capítulo anterior, estabelecemos o que são Large Language Models e como eles se tornaram a base para sistemas de IA cada vez mais sofisticados. Agora, precisamos enfrentar duas perguntas fundamentais: como um modelo que "apenas" prevê o próximo token consegue raciocinar sobre problemas complexos? E como ele consegue "usar" ferramentas externas se não passa de uma função matemática que mapeia sequências de tokens em distribuições de probabilidade?

As aspas nos verbos acima são intencionais. A tese central deste capítulo é dupla:

  1. Raciocínio é geração de tokens. Quando um LLM "pensa", ele está gerando tokens intermediários que condicionam a distribuição de probabilidade dos tokens seguintes, melhorando a qualidade da resposta final.
  2. Tool calling é sinalização, não execução. Quando um LLM "usa" uma calculadora ou consulta um banco de dados, ele está emitindo uma estrutura de dados que descreve a chamada. Quem executa é outro componente — o orquestrador.

Compreender essas duas mecânicas é pré-requisito para tudo que virá nos capítulos seguintes sobre agentes. Um agente, no fundo, é um loop que alterna entre raciocínio (geração de tokens) e ação (tool calling orquestrado). Sem clareza sobre o que acontece em cada etapa, é impossível projetar sistemas robustos.

2.1 O Raciocínio é "Só" Geração de Tokens

2.1.1 Autoregressive Decoding: A Mecânica Fundamental

Todo LLM moderno baseado na arquitetura Transformer decoder opera da mesma forma: dado uma sequência de tokens de entrada, o modelo produz uma distribuição de probabilidade sobre o vocabulário para o próximo token. Esse token é selecionado (via sampling ou greedy decoding), concatenado à sequência, e o processo se repete.

Não há um módulo separado de "raciocínio". Não existe uma CPU interna que processa lógica. Tudo o que o modelo produz — seja uma piada, uma prova matemática ou a decisão de chamar uma ferramenta — emerge do mesmo processo autoregressivo de geração token a token.

Definição

Autoregressive decoding: Processo no qual o modelo gera uma sequência de saída um token por vez, onde cada novo token é condicionado em todos os tokens anteriores (tanto do input quanto dos já gerados). É o mecanismo único e universal pelo qual LLMs produzem qualquer tipo de output.

2.1.2 Chain-of-Thought: Raciocínio como Subproduto

Em 2022, Wei et al. demonstraram algo notável no paper "Chain-of-Thought Prompting Elicits Reasoning in Large Language Models" (arXiv:2201.11903): quando você inclui exemplos de raciocínio passo a passo no prompt (few-shot), modelos suficientemente grandes passam a gerar passos intermediários antes da resposta final — e isso melhora drasticamente a acurácia em tarefas de raciocínio.

O resultado mais emblemático: o modelo PaLM com 540 bilhões de parâmetros, usando apenas 8 exemplos de chain-of-thought, atingiu o estado da arte no benchmark GSM8K (problemas matemáticos de ensino fundamental). Sem CoT, o mesmo modelo errava a maioria dos problemas.

Exemplo

Sem Chain-of-Thought:

Pergunta: Roger tem 5 bolas de tênis. Ele compra mais 2 latas
de 3 bolas cada. Quantas bolas ele tem agora?
Resposta: 9  ← ERRADO

Com Chain-of-Thought:

Pergunta: Roger tem 5 bolas de tênis. Ele compra mais 2 latas
de 3 bolas cada. Quantas bolas ele tem agora?
Pensamento: Roger começa com 5 bolas. Ele compra 2 latas com
3 bolas cada, ou seja, 2 × 3 = 6 bolas. No total, 5 + 6 = 11.
Resposta: 11  ← CORRETO

No segundo caso, os tokens do "Pensamento" não são decoração — eles condicionam a distribuição de probabilidade dos tokens seguintes, tornando muito mais provável que o modelo gere "11" e não "9" ou "10".

Por que isso funciona? A intuição é a seguinte: um Transformer tem profundidade fixa (número de camadas). Cada passada forward pela rede representa uma quantidade fixa de "computação". Para problemas que exigem múltiplos passos de raciocínio, uma única passada forward pode não ser suficiente. Gerar tokens intermediários permite que o modelo execute múltiplas passadas forward sobre resultados parciais, efetivamente ampliando sua capacidade computacional.

Dica

Chain-of-thought não requer fine-tuning. É uma propriedade emergente que pode ser ativada apenas com exemplos no prompt (few-shot) ou até com a instrução simples "pense passo a passo" (zero-shot CoT). Isso a torna uma das técnicas mais acessíveis e poderosas para melhorar a performance de LLMs.

Um ponto crucial observado por Wei et al.: CoT é uma propriedade emergente de escala. Modelos com menos de ~100 bilhões de parâmetros não apresentam melhoria significativa com chain-of-thought — e em alguns casos, a performance até piora. Isso sugere que a capacidade de "decompor" problemas em passos intermediários coerentes surge apenas quando o modelo atinge certo nível de capacidade.

2.1.3 Reasoning Tokens: A Institucionalização do Pensamento

A partir de 2024, os principais laboratórios passaram a institucionalizar a geração de tokens intermediários como mecanismo central de raciocínio, indo além do CoT induzido por prompt. Em 2026, o raciocínio integrado já é o padrão em todos os modelos de fronteira.

OpenAI: de o1 ao GPT-5.x

O modelo o1 (setembro de 2024) foi o primeiro a introduzir o conceito de reasoning tokens: antes de gerar a resposta visível ao usuário, o modelo produz um "scratchpad" interno. Depois vieram o o3 e o o4-mini (abril de 2025), aprofundando essa linha. O grande marco veio com o GPT-5 (maio de 2025), que unificou raciocínio e geração em um único modelo — o controle passou a ser feito via o parâmetro reasoning.effort (none, low, medium, high, xhigh). Os modelos mais recentes são o GPT-5.2 Thinking (dezembro de 2025) e o GPT-5.4 (março de 2026). Uma característica importante: reasoning tokens brutos nunca são expostos via API — apenas resumos via reasoning.summary.

Google Gemini: thinking como padrão

O Gemini 2.5 Pro/Flash introduziu thinking ativado por padrão com thinkingBudget dinâmico. Com o Gemini 3 (dezembro de 2025) e o 3.1 (fevereiro de 2026), o thinking está praticamente sempre presente, com controle via thinking_level. Os pensamentos não são retornados por padrão — é necessário usar includeThoughts: true para acessá-los.

Anthropic Claude: transparência no raciocínio

O Claude 4.5/4.6 (Opus/Sonnet) implementa Extended Thinking com Adaptive Thinking: blocos internos de tipo thinking que são retornados sumarizados por padrão — uma abordagem mais transparente que a da OpenAI. O controle é feito via o parâmetro effort (low, medium, high); o antigo budget_tokens está deprecated nos modelos 4.6. Uma evolução importante é o Interleaved Thinking, que permite ao modelo alternar entre blocos de thinking e chamadas de ferramentas — pensar, agir, pensar novamente, agir novamente.

Atenção

Reasoning tokens têm custo real. Um modelo que "pensa" por 10.000 tokens antes de responder custa proporcionalmente mais. Em produção, é essencial calibrar o budget de raciocínio: problemas simples não justificam cadeias longas de pensamento, enquanto problemas complexos se beneficiam enormemente. Esse é um trade-off fundamental entre custo/latência e qualidade.

O ponto unificador é este: independentemente da implementação, o mecanismo é o mesmo. O modelo gera tokens intermediários que servem como "memória de trabalho" temporária, condicionando a geração subsequente. Não há magia — há mais computação.

2.2 Tokens Intermediários: Por Que o Modelo "Pensa Antes de Responder"

2.2.1 A Limitação Computacional de Uma Única Passada

Para entender por que tokens intermediários são necessários, considere uma analogia. Imagine que alguém lhe peça para multiplicar 47 x 83 mentalmente, mas com uma restrição: você deve dar a resposta em no máximo um segundo, sem papel, sem calculadora. A maioria das pessoas erraria.

Agora, se permitirmos que você use um rascunho: 47 x 80 = 3.760; 47 x 3 = 141; 3.760 + 141 = 3.901. Com espaço para resultados intermediários, o problema se torna tratável.

Um LLM enfrenta uma versão análoga desse problema. Cada token gerado corresponde a uma única passada forward pela rede — um número fixo de operações matriciais. Para problemas que exigem raciocínio em múltiplas etapas, uma única passada forward simplesmente não contém computação suficiente para chegar à resposta correta.

Os tokens intermediários funcionam como o rascunho: cada um representa uma nova passada forward que pode operar sobre os resultados parciais já gerados. Em termos técnicos, eles expandem a profundidade efetiva de computação do modelo.

2.2.2 Evidência Empírica: Mais Tokens, Melhor Raciocínio

Os dados dos modelos de reasoning corroboram essa intuição. A OpenAI reportou que a acurácia do o1 melhora logaritmicamente com o budget de tokens de raciocínio. Isso significa que:

  • Permitir 100 tokens de raciocínio vs. 0 produz um ganho substancial
  • Ir de 100 para 1.000 tokens produz ganho significativo mas menor
  • Ir de 1.000 para 10.000 tokens produz ganho incremental
  • Os retornos são decrescentes, mas nunca irrelevantes para problemas complexos

Essa relação logarítmica tem implicação prática direta: em produção, é fundamental calibrar o budget de raciocínio por tipo de tarefa. Um chatbot de FAQ não precisa de 10.000 tokens de pensamento; um sistema de resolução de provas matemáticas pode se beneficiar de cada token adicional.

2.2.3 O Que Está nos Thinking Tokens?

Embora a OpenAI oculte o conteúdo dos reasoning tokens do o1, a Anthropic disponibiliza os blocos thinking do Claude via API. Analisando esses blocos, observamos padrões recorrentes:

  • Decomposição: O modelo quebra o problema em subproblemas menores
  • Verificação: O modelo frequentemente "checa" resultados parciais ("deixa eu verificar: 3 x 4 é mesmo 12...")
  • Exploração e backtracking: O modelo considera abordagens alternativas quando a primeira não funciona
  • Síntese: Ao final, o modelo consolida os resultados parciais em uma resposta coerente

Esses padrões são notavelmente similares às estratégias de resolução de problemas que humanos usam — mas é crucial lembrar: o modelo não está "decidindo" usar essas estratégias. Ele está gerando tokens que, probabilisticamente, se parecem com raciocínio humano, porque foi treinado em vastas quantidades de texto onde humanos raciocinam dessa forma.

Definição

Reasoning tokens (tokens de raciocínio): Tokens gerados pelo modelo antes da resposta final, que servem como "memória de trabalho" temporária. Eles condicionam a geração subsequente, permitindo ao modelo executar múltiplos passos de computação sobre resultados intermediários. Podem ser visíveis (Claude thinking blocks) ou ocultos (o1 reasoning).

2.3 Tool Calling: O LLM Não Executa Ferramentas

2.3.1 A Ilusão da Execução

Quando alguém diz que "o ChatGPT consultou a Wikipedia" ou "o Claude executou um código Python", está cometendo um erro conceitual fundamental. Como Martin Fowler sintetiza de forma precisa:

"The LLM does not execute these calls directly, instead it creates a data structure that describes the call, passing that to a separate program for execution."

O LLM faz exatamente uma coisa: gera tokens. Quando esses tokens formam uma estrutura JSON que descreve uma chamada de ferramenta, o modelo está sinalizando uma intenção — não executando nada. A execução é responsabilidade de um componente externo: o orquestrador.

2.3.2 O Fluxo Completo de Tool Calling

Vamos detalhar o fluxo passo a passo, usando como exemplo a API da Anthropic:

1. Definição de schemas (developer-time):

O desenvolvedor define quais ferramentas o modelo pode "usar", especificando nome, descrição e parâmetros em JSON Schema:

{
  "name": "consultar_clima",
  "description": "Retorna a temperatura atual de uma cidade",
  "input_schema": {
    "type": "object",
    "properties": {
      "cidade": {
        "type": "string",
        "description": "Nome da cidade"
      }
    },
    "required": ["cidade"]
  }
}

2. Análise do input (inference-time):

O modelo recebe a mensagem do usuário ("Qual a temperatura em São Paulo?") junto com as definições das ferramentas disponíveis.

3. Geração do JSON de chamada (o modelo gera tokens):

O modelo produz um bloco de tipo tool_use:

{
  "type": "tool_use",
  "id": "toolu_01A09q90qw90lq917835lq9",
  "name": "consultar_clima",
  "input": {
    "cidade": "São Paulo"
  }
}
Atenção

Neste ponto, nada foi executado. O modelo apenas gerou uma sequência de tokens que, quando parseada, descreve uma chamada de função. Não houve consulta a nenhuma API de clima. Não houve acesso à internet. O modelo simplesmente produziu texto estruturado.

4. Parsing e execução pelo orquestrador:

O orquestrador (seu código, o framework LangChain, o LangGraph, etc.) recebe a resposta da API, identifica o bloco tool_use, extrai o nome da função e os parâmetros, e executa a chamada real:

# O orquestrador faz isso - não o LLM
resultado = api_clima.consultar(cidade="São Paulo")
# resultado: {"temperatura": 24, "unidade": "celsius"}

5. Reinjeção do resultado:

O orquestrador envia o resultado de volta ao modelo como uma mensagem de tipo tool_result:

{
  "type": "tool_result",
  "tool_use_id": "toolu_01A09q90qw90lq917835lq9",
  "content": "{\"temperatura\": 24, \"unidade\": \"celsius\"}"
}

6. Geração da resposta final:

Agora, com o resultado da ferramenta no contexto, o modelo gera a resposta ao usuário: "A temperatura atual em São Paulo é de 24 graus celsius."

2.3.3 As Três Capacidades Aprendidas

Para que o tool calling funcione, o modelo precisa ter aprendido — durante o treinamento ou fine-tuning — três capacidades distintas:

  1. Reconhecimento de intenção (intent recognition): Identificar que a pergunta do usuário requer uma ferramenta externa, e não apenas conhecimento paramétrico.
  2. Seleção de ferramenta (tool selection): Dentre as ferramentas disponíveis, escolher a mais adequada para a tarefa.
  3. Geração de parâmetros (parameter generation): Preencher corretamente os argumentos da ferramenta com base no contexto da conversa.
Exemplo

As três capacidades em ação:

Usuário: "Quanto custa um dólar hoje?"

  1. Intent recognition: O modelo reconhece que a cotação do dólar é informação em tempo real → precisa de ferramenta.
  2. Tool selection: Dentre [consultar_clima, consultar_cotacao, enviar_email], seleciona consultar_cotacao.
  3. Parameter generation: Gera {"moeda_origem": "USD", "moeda_destino": "BRL"} extraindo informações do contexto.

Se o modelo errasse em qualquer uma das três etapas — por exemplo, tentando responder com conhecimento desatualizado (1), escolhendo consultar_clima (2), ou gerando {"moeda_origem": "EUR"} (3) — o resultado seria incorreto.

2.3.4 Structured Outputs: Garantindo JSON Válido

Um desafio prático do tool calling é que o modelo pode gerar JSON malformado — afinal, ele está apenas gerando tokens, e nada no processo autoregressivo garante que as chaves serão fechadas corretamente ou que os tipos estarão corretos.

Para mitigar isso, a OpenAI introduziu em junho de 2024 o mecanismo de Structured Outputs com strict: true, que utiliza constrained decoding para garantir que a saída do modelo obedece exatamente ao JSON Schema especificado. Na prática, isso funciona modificando as probabilidades dos tokens durante a geração: tokens que violariam o schema recebem probabilidade zero.

2.4 O Loop Observe → Think → Act

2.4.1 ReAct: Unificando Raciocínio e Ação

Em 2022, Yao et al. propuseram o framework ReAct (Reasoning + Acting, arXiv:2210.03629, ICLR 2023), que formaliza a integração de raciocínio e tool calling em um loop iterativo. O padrão é:

  1. Observe: O modelo recebe uma observação (input do usuário ou resultado de uma ferramenta)
  2. Think: O modelo gera tokens de raciocínio ("Preciso descobrir X. Para isso, posso usar a ferramenta Y com o parâmetro Z.")
  3. Act: O modelo gera uma chamada de ferramenta estruturada

O ciclo se repete quantas vezes forem necessárias até que o modelo julgue ter informação suficiente para produzir a resposta final.

Os resultados experimentais foram expressivos: +34 pontos em ALFWorld (tarefas de navegação em ambientes virtuais) e +10 pontos em WebShop (tarefas de compra online) comparados aos baselines.

Um achado particularmente importante: interleaving raciocínio e ação supera cada um isoladamente. Chain-of-thought puro (só raciocínio, sem ação) tende a alucinar — o modelo "inventa" fatos quando não tem acesso a informações reais. Ação pura (sem raciocínio) executa ferramentas de forma mecânica, sem capacidade de sintetizar resultados ou corrigir o curso.

2.4.2 Quem Faz o Quê: A Divisão de Responsabilidades

Este é um dos pontos mais importantes do capítulo. No loop ReAct, existem três atores com responsabilidades distintas:

Ator Responsabilidade Exemplo
LLM Gerar raciocínio + decidir ação + gerar JSON estruturado "Preciso consultar o clima. Vou usar consultar_clima com cidade='SP'."
Orquestrador Parsear a resposta, executar a ferramenta, reinjetar o resultado Recebe o JSON, chama a API de clima, retorna {"temp": 24} ao LLM
Ambiente Executar a ação concreta e retornar resultado O servidor da API de clima processa a requisição e retorna os dados
Atenção

O LLM nunca toca o ambiente diretamente. Existe sempre uma camada de orquestração entre o modelo e o mundo externo. Isso não é uma limitação — é um princípio de design fundamental que permite validação, logging, rate limiting, tratamento de erros e controle de acesso. Em sistemas de produção, essa separação é o que torna o sistema seguro e auditável.

2.4.3 Exemplo Concreto: Um Ciclo ReAct Completo

Considere um assistente com acesso a duas ferramentas: buscar_produto(query) e consultar_estoque(produto_id).

Turno 1 — Observe: Usuário pergunta "Vocês têm o iPhone 15 Pro em estoque?"

Turno 1 — Think (tokens gerados pelo LLM):
"O usuário quer saber sobre disponibilidade de um produto específico. Preciso primeiro buscar o produto para obter o ID, depois consultar o estoque."

Turno 1 — Act (JSON gerado pelo LLM):

{"name": "buscar_produto", "input": {"query": "iPhone 15 Pro"}}

Turno 2 — Observe: Orquestrador executa e retorna {"id": "SKU-8834", "nome": "iPhone 15 Pro 256GB"}

Turno 2 — Think:
"Encontrei o produto com ID SKU-8834. Agora preciso verificar o estoque."

Turno 2 — Act:

{"name": "consultar_estoque", "input": {"produto_id": "SKU-8834"}}

Turno 3 — Observe: Orquestrador retorna {"disponivel": true, "quantidade": 3}

Turno 3 — Think:
"O produto está disponível com 3 unidades em estoque. Posso responder ao usuário."

Turno 3 — Resposta final:
"Sim, temos o iPhone 15 Pro 256GB em estoque! No momento, há 3 unidades disponíveis."

Note que o LLM precisou de dois ciclos de tool calling para responder — e em cada ciclo, o raciocínio intermediário guiou a próxima ação. Sem o "Think", o modelo poderia ter tentado consultar o estoque sem o ID, ou parado após a busca sem verificar disponibilidade.

2.5 O Papel do Orquestrador

2.5.1 Muito Além de Um "Proxy"

É tentador pensar no orquestrador como um simples intermediário que repassa mensagens entre o LLM e as ferramentas. Na prática, ele é o componente mais crítico da arquitetura. Suas responsabilidades incluem:

Parsing da resposta do modelo:
O orquestrador precisa interpretar a saída do modelo e identificar se contém uma chamada de ferramenta, uma resposta final, ou ambos. Com structured outputs e stop reasons explícitos (como stop_reason: "tool_use" na API da Anthropic), isso se torna mais confiável, mas o orquestrador ainda precisa lidar com edge cases.

Execução real da ferramenta:
O orquestrador é quem de fato chama a API externa, executa o SQL, roda o código Python. Ele deve implementar timeouts, retries, circuit breakers e error handling adequados.

Reinjeção do resultado:
Após a execução, o orquestrador formata o resultado como uma mensagem de tipo tool_result e a injeta na conversa, permitindo que o modelo prossiga.

Controle do loop:
O orquestrador decide quando parar o loop. Sem limites, um modelo poderia ficar em loop infinito de tool calls. Na prática, implementa-se um número máximo de iterações e/ou um timeout global.

Segurança e validação:
Antes de executar uma chamada de ferramenta, o orquestrador pode (e deve) validar se os parâmetros são seguros, se o usuário tem permissão para a ação, se a ferramenta está disponível, etc.

# Pseudocódigo de um orquestrador simplificado
def run_agent(user_message, tools, max_iterations=10):
    messages = [{"role": "user", "content": user_message}]

    for i in range(max_iterations):
        response = llm.generate(messages, tools=tools)

        if response.stop_reason == "end_turn":
            return response.text  # Resposta final

        if response.stop_reason == "tool_use":
            for tool_call in response.tool_calls:
                # Validação
                if tool_call.name not in allowed_tools:
                    raise SecurityError(f"Tool não permitida: {tool_call.name}")

                # Execução real
                result = execute_tool(tool_call.name, tool_call.input)

                # Reinjeção
                messages.append({"role": "assistant", "content": response})
                messages.append({
                    "role": "user",
                    "content": [{
                        "type": "tool_result",
                        "tool_use_id": tool_call.id,
                        "content": json.dumps(result)
                    }]
                })

    raise MaxIterationsError("Loop excedeu o limite")
Dica

O orquestrador é onde frameworks como LangGraph brilham. Ele abstrai o loop de execução, o gerenciamento de estado, o parsing de respostas e a reinjeção de resultados. Nos próximos capítulos, veremos como o LangGraph modela esse loop como um grafo de estados, dando ao desenvolvedor controle granular sobre cada transição.

2.6 Juntando Tudo: Do Token ao Agente

Agora podemos ver como raciocínio e tool calling se compõem para formar a base dos agentes que estudaremos nos próximos capítulos:

  1. O modelo recebe o contexto (mensagem do usuário + ferramentas disponíveis + histórico)
  2. Gera reasoning tokens para analisar a situação e planejar
  3. Decide se precisa de informação externa ou se pode responder diretamente
  4. Se precisa de ferramenta: gera tokens que formam um JSON de tool call
  5. O orquestrador parseia, valida e executa
  6. O resultado é reinjetado no contexto
  7. O modelo analisa o novo contexto e decide o próximo passo
  8. O ciclo se repete até a resposta final

Cada etapa desse fluxo é geração de tokens. Cada decisão — "preciso de uma ferramenta?", "qual ferramenta?", "com quais parâmetros?", "tenho informação suficiente para responder?" — emerge do mesmo processo autoregressivo. O que diferencia um agente de um chatbot simples é o loop de orquestração que permite múltiplos ciclos de raciocínio e ação.

Resumo do Capítulo

Este capítulo desmistificou as duas mecânicas fundamentais dos agentes:

  1. Raciocínio é geração de tokens. Não há módulo separado de "pensamento" — o modelo gera tokens intermediários que condicionam a resposta final. Mais tokens de raciocínio = mais computação efetiva, com retornos logarítmicos.
  2. Chain-of-Thought é propriedade emergente que surge em modelos com 100B+ parâmetros. Pode ser ativada por prompt ("pense passo a passo") sem fine-tuning.
  3. Reasoning tokens foram institucionalizados pelos principais provedores: GPT-5.x (ocultos, resumo via API), Claude (sumarizados por padrão), Gemini (opt-in via includeThoughts).
  4. Tool calling é sinalização, não execução. O LLM gera um JSON descrevendo a intenção. O orquestrador parseia, valida, executa e reinjeta o resultado.
  5. ReAct (Observe → Think → Act) formaliza o loop de raciocínio + ação. Interleaving supera cada um isoladamente.
  6. O orquestrador é o componente crítico — responsável por parsing, execução real, reinjeção, controle de loop e segurança. É engenharia de software clássica.

A grande lição: não há magia. LLMs são máquinas de gerar tokens — extraordinariamente sofisticadas, mas mecanicamente simples. Quando "raciocinam", geram tokens intermediários. Quando "agem", geram JSON.

No próximo capítulo, veremos como o contexto — a informação que alimenta o modelo — é o recurso mais crítico de um agente, e como engenhrá-lo corretamente.
